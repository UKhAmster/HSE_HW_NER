## 1. Схема базы данных

### 1.1 Предметная область

База данных **university_db** моделирует систему учёта студентов университета и включает четыре коллекции:

| Коллекция     | Описание                         | Кол-во документов |

## 2. Реализация шардинга

### 2.1 Архитектура кластера

Кластер развёрнут через Docker Compose и состоит из:

```
┌─────────────────────────────────────────────────────────────┐
│                        mongos (router)                       │
│                      localhost:27017                          │
└──────────────┬───────────────────────────┬──────────────────┘
               │                           │
    ┌──────────▼──────────┐     ┌──────────▼──────────┐
    │   Shard 1 (shard1rs)│     │   Shard 2 (shard2rs)│
    │   ┌──────────────┐  │     │   ┌──────────────┐  │
    │   │  shard1svr1   │  │     │   │  shard2svr1   │  │
    │   │  (PRIMARY)    │  │     │   │  (PRIMARY)    │  │
    │   ├──────────────┤  │     │   ├──────────────┤  │
    │   │  shard1svr2   │  │     │   │  shard2svr2   │  │
    │   │  (SECONDARY)  │  │     │   │  (SECONDARY)  │  │
    │   ├──────────────┤  │     │   ├──────────────┤  │
    │   │  shard1svr3   │  │     │   │  shard2svr3   │  │
    │   │  (SECONDARY)  │  │     │   │  (SECONDARY)  │  │
    │   └──────────────┘  │     │   └──────────────┘  │
    └─────────────────────┘     └─────────────────────┘
               │                           │
    ┌──────────▼───────────────────────────▼──────────┐
    │          Config Server RS (cfgrs)                 │
    │  configsvr1 · configsvr2 · configsvr3             │
    └──────────────────────────────────────────────────┘
```

**Всего контейнеров: 10**
- 3 Config Server (replica set `cfgrs`)
- 3 Shard 1 (replica set `shard1rs`)
- 3 Shard 2 (replica set `shard2rs`)
- 1 Mongos (router)

### 2.2 Выбор ключа шардирования

Для всех коллекций выбрана **хешированная** стратегия шардирования:

| Коллекция     | Shard Key            | Стратегия |
|---------------|----------------------|-----------|
| `students`    | `{ student_id: "hashed" }` | Hashed    |
| `courses`     | `{ course_id: "hashed" }`  | Hashed    |
| `enrollments` | `{ student_id: "hashed" }` | Hashed    |
| `grades`      | `{ student_id: "hashed" }` | Hashed    |

### 2.3 Обоснование выбора

**Почему хешированный шардинг?**

1. **Равномерное распределение.** UUID-подобные `student_id` / `course_id` не имеют естественного порядка. Хешированный ключ гарантирует равномерное распределение данных между шардами, избегая «горячих точек» (hot spots).

2. **Высокая кардинальность.** Каждый `student_id` уникален → MongoDB распределяет чанки максимально равномерно.

3. **Эффективные точечные запросы.** Запрос `find({ student_id: X })` направляется напрямую на один шард (targeted query), без scatter-gather.

4. **Консистентность по `student_id`.** Все данные одного студента (`enrollments`, `grades`) хранятся на одном шарде, что ускоряет `$lookup` операции.

**Компромиссы:**

- Range-запросы по `student_id` становятся scatter-gather (но такие запросы не типичны для UUID).
- Запросы по `faculty` или `last_name` — scatter-gather. Это решается вторичными индексами на каждом шарде.

### 2.4 Распределение данных

После загрузки 50 000 студентов данные распределяются равномерно:

```
students:     50,000 документов
  └─ shard1rs: ~25,000 (50.0%)
  └─ shard2rs: ~25,000 (50.0%)

courses:      500 документов
  └─ shard1rs: ~250 (50.0%)
  └─ shard2rs: ~250 (50.0%)

enrollments:  ~275,000 документов
  └─ shard1rs: ~137,500 (50.0%)
  └─ shard2rs: ~137,500 (50.0%)

grades:       ~247,000 документов
  └─ shard1rs: ~123,500 (50.0%)
  └─ shard2rs: ~123,500 (50.0%)
```

## 4. Нагрузочное тестирование

### 4.1 Методология

Тестирование проводилось скриптом `load_test.py` с использованием **7 типов нагрузки**:

| # | Тест                     | Описание                                         | Операций |
|---|--------------------------|--------------------------------------------------|----------|
| 1 | Bulk INSERT              | Пакетная вставка по 500 документов               | 5 000    |
| 2 | READ (shard key)         | Точечный поиск по `student_id` (targeted)        | 5 000    |
| 3 | READ (scatter-gather)    | Поиск по `faculty` (все шарды)                   | 5 000    |
| 4 | UPDATE (shard key)       | Обновление по `student_id` (targeted)            | 5 000    |
| 5 | Aggregation pipeline     | Агрегация с `$group` + `$sort` + `$limit`        | 50       |
| 6 | Mixed (70R/20U/10I)      | Смешанная нагрузка                               | 5 000    |
| 7 | Concurrent scaling       | Конкурентная нагрузка (1–16 потоков)              | 2 000    |

### 4.3 Визуализация

Графики сохраняются в директории `results/`:

- `throughput_chart.png` — пропускная способность по типам операций
- `latency_chart.png` — латентность (AVG / P50 / P95 / P99)
- `latency_distribution.png` — гистограммы распределения латентности
- `concurrent_scaling.png` — масштабирование при конкурентной нагрузке


### 4.4 Анализ результатов

**Ключевые выводы:**

1. **Targeted vs Scatter-Gather.** Запросы по shard key (`student_id`) значительно быстрее, чем scatter-gather запросы по `faculty`, так как направляются только на один шард.

2. **Линейное масштабирование.** При увеличении числа потоков общая пропускная способность растёт почти линейно, что подтверждает эффективность горизонтального масштабирования.

3. **Хвостовая латентность.** P99 латентность в 3–5 раз выше средней, что типично для распределённых систем и связано с сетевыми задержками и балансировкой нагрузки mongos.

4. **Агрегации.** Самый медленный тип запросов, так как требует обработки данных на всех шардах с последующим слиянием результатов на mongos.

## 5. Запуск проекта

### 5.1 Требования

- Docker + Docker Compose
- Python 3.10+
- ~2 GB RAM для кластера

### 5.2 Инструкция

```bash
# 1. Запуск кластера
docker-compose up -d

# 2. Инициализация шардинга
chmod +x init_cluster.sh
./init_cluster.sh

# 3. Установка зависимостей Python
pip install -r requirements.txt

# 4. Загрузка тестовых данных (50 000 студентов)
python seed_data.py

# 5. Запуск CLI
python cli.py

# 6. Запуск веб-интерфейса (опционально)
python web_app.py

# 7. Нагрузочное тестирование
python load_test.py

# 8. Остановка кластера
docker-compose down -v
```
